# Cognitive Controller — Architectural Decisions Record

**Project:** Cognitive Controller for OpenClaw  
**Status:** Pre-Implementation (Phase 0)  
**Last Updated:** 2026-03-03

---

## Context

OpenClaw is less than 90 days old. This is the first cognitive layer plugin being built for the platform. The core problem: as agent sessions accumulate context (skills, steering files, notes, history), the working memory available for reasoning shrinks. When context exceeds 50% of the window, agents become "reactive zombies" — responding but not reasoning effectively.

The Cognitive Controller addresses this through structured memory management, bounded snapshots, and intelligent context curation.

---

## Decision 1: Structured Snapshots Over Prose Summaries

**Status:** ✅ Decided  
**Date:** 2026-02-28

### Decision
Use structured, field-based snapshots (9 fixed fields, ~500 tokens) instead of prose summaries.

### Rationale
- Prose summaries suffer from "JPEG artifacts" — summary-of-summary degradation where LLMs hallucinate and smooth away critical details
- Structured fields prevent re-interpretation drift across inference runs
- Constraints can be preserved verbatim (never paraphrased)
- Compact and token-efficient
- Auditable with explicit source tracing

### Alternatives Considered
- **Prose summary snapshots:** Simple to generate but high drift risk, constraints get paraphrased, hard to audit
- **Hybrid approach:** Rejected as unnecessarily complex for v1

### Consequences
- Requires reranker assembly logic (more upfront complexity)
- Prevents hallucinated context drift
- Enables deterministic recovery points
- Supports automation with crisp boolean signals

## Decision 2: Atomic Facts + Verbatim Constraints

**Status:** ✅ Decided  
**Date:** 2026-02-28

### Decision
Store facts as atomic, timestamped, source-linked records. Store constraints verbatim without any paraphrasing.

### Rationale
- Prevents "telephone game" distortion where "Never modify production DB without approval" becomes "Avoid production changes" becomes "Don't touch prod"
- Facts remain verifiable (tool output, user statement, observable event)
- No interpretation layer that could introduce hallucinations
- Single narrative line only in `objective_now` field

### Rules Established
1. **Atomic Facts:** Small, timestamped, source-linked, verifiable
2. **Verbatim Constraints:** Copied exactly as stated, never improved or paraphrased
3. **Single Narrative Line:** Only `objective_now` provides prose flow; everything else is structured

### Consequences
- Slightly more storage overhead (metadata per fact)
- Eliminates memory smoothing hallucinations
- Enables precise audit trails
- Facts and constraints remain unchanged across sessions

## Decision 3: SQLite for Operational Memory, Markdown for Archival

**Status:** ✅ Decided  
**Date:** 2026-02-28

### Decision
Use SQLite for operational memory (events, snapshots) and Markdown for archival roll-ups. Never mix these responsibilities.

### Rationale
- **SQLite snapshots:** Raw, high-fidelity recovery points, bounded (1-10 per thread), never summarized
- **Markdown vault:** Only curated roll-ups (daily/weekly summaries), decisions + stable knowledge, human-readable
- Separation keeps vault clean and snapshots focused
- SQLite provides ACID guarantees for operational data
- Markdown provides human-readable long-term knowledge

### Storage Boundaries
| Concern | Storage | Characteristics |
|---------|---------|-----------------|
| Operational recovery | SQLite | Raw, bounded, overwritten |
| Long-term knowledge | Markdown | Curated, permanent, human-readable |
| Event logs | SQLite | Append-only, high-fidelity |
| Decisions/ADRs | Markdown | Immutable, traceable |

### Consequences
- Two storage systems to maintain
- Clear separation of concerns
- Prevents "Markdown sprawl" in operational memory
- Enables efficient recovery without parsing text files

## Decision 4: Canonical vs Editorial Roll-Ups

**Status:** ✅ Decided (Canonical first, Editorial deferred)  
**Date:** 2026-02-28

### Decision
Implement two-stage roll-up pipeline: deterministic canonical roll-up (truth artifact) and optional LLM-enhanced editorial roll-up (readability artifact).

### Rationale
- **Canonical Roll-Up:** Generated by script only, uses SQLite events + snapshot + constraints, includes event IDs and atomic facts verbatim, deterministic and stable
- **Editorial Roll-Up:** Optional LLM pass for readability, must cite canonical items, can be regenerated without loss
- Editorial never becomes input to next roll-up (prevents summary-of-summary)

### Implementation Priority
- **Phase 1:** Canonical roll-up only
- **Phase 2+:** Editorial roll-up (optional enhancement)

### Guardrails for Editorial (when implemented)
- **Rule A:** No new facts (every sentence must trace to fact_id/decision_id/constraint)
- **Rule B:** Immutable constraints (copied verbatim)
- **Rule C:** Drift detection (content hash validation)

### Consequences
- More plumbing complexity (worth it for accuracy)
- Deterministic truth survives model quirks
- Readability layer is optional and regenerable
- No recursive compression risk

## Decision 5: Bounded Snapshot Cache (1-10 per Thread)

**Status:** ✅ Decided  
**Date:** 2026-02-28

### Decision
Maintain 1-10 snapshots per thread in SQLite. Purge older snapshots beyond this bound.

### Rationale
- Prevents unbounded growth
- Provides sufficient recovery points for rollback
- Forces consolidation into roll-ups for long-term retention
- Keeps operational memory bounded and predictable

### Retention Policy
**Keep:**
- Last N snapshots (always retained)
- Snapshots younger than 7-14 days

**Purge:**
- Older snapshots beyond age threshold
- Maintain 1-10 snapshots per thread maximum

### Consequences
- Requires periodic cleanup job
- Forces discipline in what gets promoted to roll-ups
- Prevents SQLite database from growing unbounded
- Enables fast recovery without scanning large datasets

## Decision 6: 500 Token Snapshot Budget

**Status:** ✅ Decided  
**Date:** 2026-02-28

### Decision
Target 300-500 token budget for snapshots, with structured field allocation.

### Token Allocation
- Objective + status: ~20 tokens
- Threads (5-7 rows): ~100 tokens
- Constraints (5-10 lines): ~80 tokens
- Recent facts (10-15 bullets): ~150 tokens
- Decisions (2-3 key): ~100 tokens
- Questions + tool_state + actions: ~50 tokens
- **Total:** ~500 tokens

### Rationale
- Fits comfortably in context without dominating it
- Structured format is more token-efficient than prose
- Leaves room for actual task work in the context window
- Predictable and bounded

### Consequences
- Requires reranker to prioritize and trim
- Forces selection of highest-signal information
- Enables consistent context injection
- Prevents context window bloat

## Decision 7: Reranker Scoring Priority

**Status:** ✅ Decided  
**Date:** 2026-02-28

### Decision
Score snapshot candidates in priority order: Risk → Recency → Task Relevance → Progress Signal.

### Rationale
- **Risk first:** Constraints and errors get boosted (safety first)
- **Recency:** Newer events beat older ones (temporal relevance)
- **Task relevance:** Does this relate to current objective?
- **Progress signal:** Moving/active threads ranked higher than stalled ones

### Assembly Process
1. Extract candidates from event stream
2. Score each candidate by priority order
3. Pack fields until token budget reached
4. Validate (all constraints present, objective clear, next step defined)

### Consequences
- Safety-first approach (constraints never dropped)
- Recent context prioritized over stale information
- Active work surfaces before completed work
- Deterministic and auditable ranking

## Decision 8: Confidence Gradient for Escalation

**Status:** ✅ Decided  
**Date:** 2026-02-28

### Decision
Use softmax confidence gradient (0-1) to drive escalation decisions without changing model capabilities.

### Confidence Signals
- **Confidence ↑:** Consistent evidence, moving threads, progress signals
- **Confidence ↓:** Stalled work, conflicting signals, unresolved blockers

### Effect on Routing
- High confidence (0.8-1.0): Let thread run, balanced execution
- Medium confidence (0.4-0.7): Monitor closely
- Low confidence (0.0-0.3): Escalate or pause, push-hard mode

### Rationale
- Provides quantitative signal for when to escalate
- Doesn't require model capability changes
- Enables automated triage decisions
- Surfaces stalled threads for operator attention

### Consequences
- Requires confidence scoring logic in reranker
- Enables elastic resolver to make escalation decisions
- Provides operator visibility into thread health
- May need tuning after Phase 0 deployment

## Decision 9: Event-Driven Heartbeat (No Synthetic Timers)

**Status:** ✅ Decided  
**Date:** 2026-02-28

### Decision
Implement event-driven heartbeat triggered by real activity, not synthetic timers.

### Rationale
- Reduces overhead (no polling)
- Heartbeat reflects actual system activity
- Simpler implementation
- More accurate health signal

### Heartbeat Triggers
- Tier A events (user messages, tool results, state transitions)
- Stale signal if no events for 5 minutes

### Consequences
- Heartbeat frequency varies with activity
- No guaranteed heartbeat interval
- Requires stale detection for idle periods
- Lower system overhead

## Decision 10: TypeScript Plugin (No Core Modifications)

**Status:** ✅ Decided  
**Date:** 2026-02-28

### Decision
Implement Cognitive Controller as a standard TypeScript plugin that requires no modifications to OpenClaw core.

### Rationale
- OpenClaw is less than 90 days old (platform stability)
- Plugin architecture provides isolation
- Easier to deploy and maintain
- Doesn't risk breaking core functionality
- Can be disabled/removed without affecting OpenClaw

### Consequences
- Must work within plugin API constraints
- May have limited access to internal OpenClaw state
- Requires Windows-safe deployment paths
- Graceful degradation if plugin fails
